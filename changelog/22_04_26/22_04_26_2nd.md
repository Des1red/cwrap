# cwrap — Changelog
`2026-04-22` `v2.4.0`

## Overview
Exploit engine foundation. Report parsing, finding queue, probe stubs, CSRF
capture, and full constant coverage across the report writer/parser boundary.

---

## Exploit Engine

### New: `cwrap exploit <report>`
New top-level command that reads a recon report and runs targeted exploit
probes against all actionable findings automatically.

```
cwrap exploit report/localhost-9001_2026-04-22.report
cwrap exploit report/localhost-9001_2026-04-22.report --safe
cwrap exploit report/localhost-9001_2026-04-22.report --debug
```

No subcommands — exploit does everything it can from the report findings.

### New: `internal/exploit` package
- `exploit.go` — entry point, `Run(path, flags)`, prints engine header and iterates queue
- `report.go` — `loadReport`, `validateReportPath`, `parseMapInts`, report structs
- `queue.go` — `buildQueue`, `ExploitItem` interface
- `probes.go` — `IDORProbe`, `OwnershipProbe`, `CredentiallessProbe` (stubs)

### New: `ExploitExecutor` in runner
```go
type ExploitExecutor struct{}
func (ExploitExecutor) Run(req model.Request) error {
    return exploit.Run(req.FilePath, req.Flags)
}
```

### New: `ExploitHandler` in intent
Wired into `Parse`, `Resolve`, `cmd.Execute`. Uses `FilePath` instead of
`URL` — the third argument is treated as a report path, not a network target.

## Report Parser

### New: `validateReportPath`
Checks path is non-empty, has `.report` extension, exists, and is not a
directory. Returns descriptive errors for each failure mode.

### New: Full identity parsing in `loadReport`
Parses identity lines from report into `ReportIdentity`:
- Name, tags, role, uid
- Cookie names
- CSRF token, header name, cookie name

### New: Access/denied map parsing
`parseMapInts` parses `map[identity:count ...]` format from report lines
into `map[string]int`. Used for `IdentityAccess` and `IdentityDenied` on
`ReportParam`.

### New: `ReportIdentity` struct
```go
type ReportIdentity struct {
    Name           string
    Tags           []string
    Cookies        []string
    Role           string
    UserID         string
    CSRFToken      string
    CSRFHeader     string
    CSRFCookieName string
}
```

### New: CSRF fields on `ReportParam`
`IdentityAccess` and `IdentityDenied` maps added to `ReportParam` so exploit
probes know which identities have access to which parameters.

## Queue

### New: `buildQueue` with deduplication
- `seenParam` map deduplicates by `entity URL + param name`
- `PossibleIDOR`/`SuspectIDOR` takes priority over `OwnershipBoundary`
  for the same param via `continue`
- `seenCredless` collapses all credentialless findings to one probe

## CSRF Capture

### New: CSRF token/header/cookie capture in `extractIdentity`
```go
// response headers
if token := resp.Header.Get("X-CSRF-Token"); token != "" {
    id.HasCSRF = true
    id.CSRFToken = token
    id.CSRFHeader = "X-CSRF-Token"
}

// Set-Cookie scan
for _, c := range resp.Cookies() {
    ln := strings.ToLower(c.Name)
    if strings.Contains(ln, "csrf") || strings.Contains(ln, "xsrf") {
        id.HasCSRF = true
        id.CSRFToken = c.Value
        id.CSRFCookieName = c.Name
    }
}
```

Cookie name is captured as-is from the server — no assumptions about naming.
Exploit knows exactly which cookie to send back and what value to use.

### New: CSRF fields on `knowledge.Identity`
```go
HasCSRF        bool
CSRFToken      string
CSRFHeader     string
CSRFCookieName string
```

### CSRF preserved across probes
```go
if id.CSRFToken == "" && prev.CSRFToken != "" {
    id.CSRFToken = prev.CSRFToken
    id.HasCSRF = true
}
```

## Notes on CSRF token reuse
CSRF tokens captured in the report are session-scoped and already stale by
the time exploit runs. The report values serve as **capability detection**
— exploit knows the endpoint issues CSRF tokens and which cookie name to
look for. When a probe needs a valid CSRF token it must fetch a fresh session
first and extract the new token before making the exploit request.
