# cwrap Changelog

---

## [v2.7.1] — 2026-04-29

### Fixed

#### Probe explosion on organic query-param endpoints (`expandDiscovery`)
- **Root cause**: `expandDiscovery` Phase 2 (heuristic `id/page/limit/q` injection) was running on entities
  whose URLs already contained query params (e.g. `/?id=1` discovered from PrivateBin HTML).
  This generated new param-variant entities which each received full expansion — method sweeps,
  identity probes, further discovery — causing combinatorial probe explosion.
- **Fix**: Added `urlHasQuery` guard in `expandDiscovery`. Phase 2 now returns early if
  `bodyProbed || urlHasQuery`. Organic query-param endpoints (e.g. `/api/orders?user_id=1`)
  are handled by Phase 1 (body-derived params) before the guard triggers.
- **Files**: `internal/recon/behavior/expand.go`

#### Param-variant entities expanding into full probes (`Expand`)
- **Root cause**: Entities created by probe-injected query params were not reliably
  blocked from expansion, leading to secondary method sweeps and identity probes
  on probe-generated URLs.
- **Fix**: `IsSPAFallback` and `IsParamVariant` early-return guards added at top of `Expand()`.
- **Files**: `internal/recon/behavior/expand.go`

#### `OPTIONS 204` incorrectly registering as confirmed method (`runQueuedProbes`)
- **Root cause**: `AddMethod` was called for `OPTIONS` responses. Since CORS middleware
  responds to OPTIONS for any path regardless of whether the real method works,
  every discovered endpoint showed `[OPTIONS]` in its methods list — providing no
  meaningful signal.
- **Fix**: Gated `AddMethod` call with `probe.Method != "OPTIONS"`. OPTIONS still executes
  for CORS analysis but is never recorded as a confirmed method.
- **Files**: `internal/recon/behavior/run.go`

### Added

#### SPA fallback detection (`IsSPAFallback`)
- **Problem**: Single-page applications return 200 + identical HTML for every unknown
  route (React/Vite catchall). cwrap was treating each of these as real endpoints,
  inflating entity counts, public endpoint lists, and signal counts with noise.
  On `mcp.kobold.htb` this produced 44 entities and 7 false public endpoints where
  only 5 were real.
- **Implementation**:
  - `EntityState.IsSPAFallback bool` added to knowledge model.
  - During probe execution (`runQueuedProbes`), each 200 response is fingerprinted
    (SHA-256 of status+body) and compared to `e.baseFP` (root baseline fingerprint
    captured in `Run()`). If they match and target is not root, entity is marked
    `IsSPAFallback = true`.
  - `Expand()` returns immediately for SPA fallback entities — no method sweeps,
    no discovery, no identity probes.
  - `stripSPAShellSignals()` extended: SPA fallback entities have all signals stripped
    (`PublicAccess`, `AuthBoundary`, `RoleBoundary`, `AdminSurface`,
    `CredentiallessTokenIssuance`, `ObjectOwnership`, `PossibleIDOR`) and
    `AuthLikely` cleared.
  - Reporter (`writeEntityDetails`, `writeGlobalStats`, `printSummary`) filters out
    SPA fallback entities from entity listings, signal counts, public endpoint lists,
    and summary statistics.
- **Validation**: `mcp.kobold.htb` public endpoints reduced from 7 → 5 (false `/authorize`
  and `/oauth/callback/debug` correctly excluded). `localhost:8081` public endpoints
  reduced from 11 → 1, admin surfaces from 4 → 0 (all were SPA fallback routes).
- **Files**: `internal/recon/knowledge/entity.go`, `internal/recon/behavior/run.go`,
  `internal/recon/behavior/expand.go`, `internal/recon/behavior/cleanup.go`,
  `internal/report/full.go`, `internal/report/summary.go`
