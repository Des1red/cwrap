# cwrap recon — JS file fetching, report cleanup & SPA support
`20/04/26` `v2.2.2`

## Overview

Three fixes driven by testing against OWASP Juice Shop — a modern Angular
SPA with a REST API. JS file fetching was added to enable endpoint discovery
from bundled JS, param discovery was suppressed on static JS files to reduce
noise, and the discovery tree was rewritten to handle circular JS references
cleanly.

---

## Fix 1 — JS file fetching

### Problem
`<script src="">` tags were added as graph edges but the JS files were never
fetched. `jsintel.Learn` only ran on inline JS. For SPAs like Juice Shop where
all endpoints are defined inside `main.js`, the engine discovered nothing
beyond the root HTML page.

### Changes

**`internal/recon/http/html.go`** — script src handler:
- When a `<script src="">` is found and normalized, push a GET probe via
  `e.k.PushProbe` with `Reason: knowledge.ReasonJSFetch`, priority 70
- When the probe executes, `learn` is called on the response, which calls
  `jsintel.Learn` on the JS content
- This triggered discovery of 14 API endpoints from Juice Shop's `main.js`

**`internal/recon/knowledge/constants.go`**
- Added `ReasonJSFetch = "js-fetch"`

---

## Fix 2 — Skip param discovery on JS file entities

### Problem
JS file entities (`main.js`, `polyfills.js`, `scripts.js`) were getting
`expandDiscovery` run on them — injecting `id=1`, `user=1` etc. as query
params. These produce meaningless probes against static files and generate
false positive "Unauthenticated access observed" findings since static files
return 200 to everyone.

### Changes

**`behavior/expand.go`**
- Added `isJSEntity(ent *knowledge.Entity) bool` — checks URL suffix for `.js`
- In `Expand`, JS entities skip `expandDiscovery`, `expandPathIDs`,
  `expandMutation`, `expandEnumeration`
- JS entities still run `expandMethods` and `expandIdentity` — checking if a
  JS file requires auth is still valid intelligence

---

## Fix 3 — Discovery tree global visited dedup

### Problem
The discovery tree walker used a path-local `stack` to detect cycles. This
prevented infinite recursion but allowed the same node to appear as a full
subtree multiple times via different paths. With Juice Shop's 3 JS files each
referencing each other, the tree expanded to hundreds of lines of redundant
subtrees.

### Changes

**`report/full.go`** — `writeDiscoveryTree`:
- Replaced path-local `stack map[string]bool` with global `visited
  map[string]bool`
- Each node now appears exactly once in the tree — subsequent references
  show `(seen)` instead of expanding again
- Walk function simplified — no more `copySet` per recursion level

---

## Fix 4 — Entity ordering by signal count

### Problem
Entities were printed alphabetically. For targets with many entities, the most
interesting ones (high signal count) were buried between noise entities.

### Changes

**`report/full.go`**
- Added `entityURLsBySignalCount(k *knowledge.Knowledge) []string`
- Added `signalCount(ent *knowledge.Entity) int`
- `writeEntityDetails` now iterates entities sorted by signal count descending,
  with alphabetical fallback for ties
- API endpoints with 5 signals appear before JS files with 2 signals

---

## Results on Juice Shop

Before: 1 entity, 3 edges, 0 meaningful findings

After:  19 entities, 33 edges, 10 role boundaries, 53 JS leaks, 225 endpoints
        discovered from main.js
