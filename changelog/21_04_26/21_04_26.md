# cwrap recon — structural analysis and discovery improvements
`21/04/26` `v2.3.2`

## Overview
Two improvements targeting IDOR detection accuracy and param discovery
quality. Canonicalization now properly normalizes response bodies for
structural comparison instead of raw byte diffing, eliminating false
positives from dynamic values. Discovery now derives probe candidates from
actual response schema rather than guessing from path segments and a
hardcoded list, and the HTTP engine can now see JSON response bodies for
the first time.

---

## Improvement 1 — Proper response canonicalization

### Problem
`analyzeParamBehavior` and `analyzeIDOR` compare response bodies to detect
structural changes across different param values. Previously:

- The **API engine** `Canonicalize` called `normalizeJSONWithParam` which
  only replaced the value of one specific key (`param`) with `"<param>"`,
  leaving all other values intact — names, timestamps, IDs, tokens. Two
  responses with identical schema but different data would appear structurally
  different, producing widespread false `ObjectAccess` signals.
- The **HTTP engine** `Canonicalize` was a no-op — `return body, nil`. All
  comparison was raw byte diffing. Any dynamic content caused false diffs.

### Fix
Created a shared `internal/recon/canonicalize` package with three functions:

- `JSON(data, param)` — replaces **all** leaf values with typed placeholders
  (`<string>`, `<number>`, `<bool>`, `nil`). Arrays collapsed to first
  element — list length is not a schema difference. Two responses with the
  same schema but different data now compare equal. `ObjectAccess` only fires
  when the key set, types, or nesting genuinely differ.
- `HTML(data)` — strips script/style content, attribute values, and text
  node content, keeping only tag structure and attribute names. Two pages
  with the same DOM structure but different dynamic content compare equal.
- `StripNumbers(b)` — fallback for unknown content types. Replaces digits
  with `#` to remove numeric noise.

The HTTP engine `Canonicalize` now detects content type from the body
itself (`looksLikeJSONBody`, `looksLikeHTMLBody`) and delegates to the
appropriate function. The API engine `Canonicalize` delegates to
`canonicalize.JSON`. Both engines share the same normalization logic.

### Changes

**`internal/recon/canonicalize/canonicalize.go`** — new package
- `JSON(data []byte, param string) ([]byte, error)`
- `HTML(data []byte) []byte`
- `StripNumbers(b []byte) []byte`

**`internal/recon/http/interpreter.go`**
- `Canonicalize`: replaced no-op with content-aware dispatch to
  `canonicalize.JSON`, `canonicalize.HTML`, or `canonicalize.StripNumbers`
- Added `looksLikeJSONBody` and `looksLikeHTMLBody` detection helpers

**`internal/recon/api/interpreter.go`**
- `Canonicalize`: replaced local `normalizeJSONWithParam` call with
  `canonicalize.JSON`

**`internal/recon/api/normalize.go`**
- `normalizeJSONWithParam` and `stripParamValue` deleted — dead code
  replaced by shared package

## Improvement 2 — Body-derived param discovery

### Problem
`expandDiscovery` was the primary mechanism for discovering query params on
endpoints. It had two strategies:

1. Split the URL path into segments and probe each as a query param
   (e.g. `/api/users` → `?api=1`, `?users=1`)
2. Fall back to a hardcoded list: `["id", "user", "page", "q"]`

Both strategies are guesses. The actual query interface of an endpoint is
not inferable from its path segments. Meanwhile, JSON response bodies and
HTML forms already contain ground-truth schema information in `ent.Params`
(registered as `ParamJSON` and `ParamForm`) — but `expandDiscovery` was
ignoring all of it.

Additionally, the HTTP engine's `learn` function never called any JSON
extraction, making it completely blind to JSON response bodies. All JSON
param discovery was only happening in the API engine.

### Fix

**Shared JSON extraction (`internal/recon/jsonintel`):**
Created a shared `jsonintel` package that both engines now call for JSON
responses. Walks the response body up to 2 levels deep (top-level keys and
one level down — deeper nesting is data content, not query params). Only
walks the first array element. Skips `reflectionKeys` (httpbin-style
transport echo fields and error envelope keys). Registers discovered keys
as `ParamJSON` and classifies them via `paramintel`.

**HTTP engine JSON extraction:**
`http/learn.go` now calls `jsonintel.ExtractParams` for JSON responses.
Previously the HTTP engine registered zero params from JSON bodies.

**Body-derived discovery in `expandDiscovery`:**
Phase 1 now probes `ParamJSON` and `ParamForm` params that the response
body already revealed, using type-appropriate values via `discoveryValuesFor`:
- `IDLike` params → `"1"`, `"2"`, `"0"`
- Pagination (`page`, `limit`, `offset`, etc.) → realistic values
- Boolean flags (`is_*`, `enabled`, `active`, etc.) → `"true"`, `"false"`
- Sort params → `"asc"`, `"desc"`
- Search params → `"test"`, `""`
- Generic → `"1"`

Phase 2 (path segment heuristics + minimal hardcoded fallback) only runs if
Phase 1 found nothing — i.e. on endpoints with no extractable body schema.
Hardcoded fallback trimmed from `["id", "user", "page", "q"]` to
`["id", "page", "limit", "q"]`.

**Item endpoint guard:**
JSON params are not probed as query filters on item endpoints (entities that
already have a `ParamPath` param). Response field names on `/api/users/1`
describe the user object — they are not query params. Collection endpoints
(`/api/users`, `/api/posts`) are not affected and still benefit from
body-derived discovery.

**2xx-only extraction:**
Both engines now only extract JSON params on 2xx responses. Error bodies
(`{"error":"not found"}`) were previously being parsed, registering `error`
as a param on every entity.

**Expanded `reflectionKeys`:**
Added error envelope keys to the skip list: `error`, `message`, `detail`,
`details`, `code`, `status`, `trace`, `errors`.

### Changes

**`internal/recon/jsonintel/jsonintel.go`** — new package
- `ExtractParams(ent, k, data)` — shared JSON param extraction
- `walkJSON` with depth limit (max 2) and first-element array walking
- Expanded `reflectionKeys`

**`internal/recon/api/extract.go`**
- `extractJSON` now delegates to `jsonintel.ExtractParams`
- Local `walkJSON`, `reflectionKeys` deleted

**`internal/recon/http/learn.go`**
- Added `jsonintel.ExtractParams` call for 2xx JSON responses

**`internal/recon/api/learn.go`**
- `extractJSON` now gated on `resp.StatusCode >= 200 && resp.StatusCode < 300`

**`behavior/expand.go`**
- `expandDiscovery` restructured into Phase 1 (body-derived) and Phase 2
  (fallback heuristics)
- Added `discoveryValuesFor(p *ParamIntel) []string` helper
- Added `hasPathParam` guard to skip JSON params on item endpoints
- Hardcoded fallback list trimmed
