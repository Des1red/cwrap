# cwrap recon — role boundary detection & identity stability
```19/04/26```
## Overview

Three additions this session: a new `SigRoleBoundary` signal that distinguishes
authenticated-but-denied from unauthenticated-denied, a direct entity-level role
boundary detector that bypasses the param map, and a ratchet fix for identity
state flickering across probe executions.

---

## Fix 1 — Role boundary signal

### Problem
`SigAuthBoundary` covered both cases: no credentials (401) and wrong role (403
from an authenticated identity). These are fundamentally different security
findings — one means "you need to log in", the other means "you're logged in but
don't have permission". Treating them identically loses precision in the report
and next steps.

### Changes

**`knowledge/signals.go`**
- Added `SigRoleBoundary` signal — fires when an authenticated identity is denied
- Added `"RoleBoundary"` to `Signal.String()`

**`behavior/analyze.go`** — updated `analyzeAuthBoundary`:
- 401 and anonymous 403 → `SigAuthBoundary` (requires `has200` — mixed access)
- 403 from identity with `SentCreds=true` → `SigRoleBoundary` (no `has200` required)
- `ObservedChanges` keys split: `"auth-wall-mixed-status"` vs `"role-wall-403-authenticated"`

**`behavior/run.go`** — added `detectRoleBoundary`:
- Runs directly off `identityStatuses` per probe, bypassing the param map
- Necessary because method probes have no `AddQuery`/`PathParams` so `storeResponse`
  stores nothing — `analyzeAuthBoundary` never sees the statuses for those probes
- Tags `SigRoleBoundary` when any identity with `SentCreds=true` gets 403

**`report/derive.go`**
- Added finding: "Role/permission boundary observed — authenticated identity denied access"
- Added next steps: vertical privilege escalation, role confusion headers, JWT claims tampering

**`report/summary.go`**
- Added `hasRoleBoundary` flag
- Added `Role boundary: yes/no` line to summary output

## Fix 2 — Identity state flickering

### Problem
`extractIdentity` builds a fresh `Identity` struct on every probe execution and
replaces the existing record via `AddIdentity`. If a probe request doesn't send
a Cookie header (e.g. certain method probes), `SentCreds` resets to `false` even
though the identity was previously established as credentialed. This caused
`detectRoleBoundary` to miss the role boundary on probes where `SentCreds`
flickered back to `false`.

Same problem affected `Role` — JWT claims extracted on one probe could be lost
on the next if the response didn't include the claim again.

### Changes

**`behavior/identity.go`** — consolidated preservation block before `ent.AddIdentity`:
- `SentCreds` is now a one-way ratchet — once `true`, never resets to `false`
- `Role` is preserved from previous record if current probe didn't extract one
- `Effective` preservation was already present — merged into the same block
- Single map lookup replaces the previous two separate lookups
