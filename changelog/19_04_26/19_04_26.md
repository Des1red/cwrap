# cwrap recon ‚Äî probe engine fixes & new detection
```19/04/26```
## Overview

Seven fixes to the probe loop, dedup system, auth detection, and a new
credentialless token issuance analyzer. All changes are confined to
`internal/recon/behavior/`, `internal/recon/knowledge/`, and `internal/recon/report/`.

---

## Fix 1 ‚Äî Probe dedup regression

### Problem
`pushProbe` was marking `ent.SeenProbes[key]` immediately for all entities
including root. The scheduler in `runQueuedProbes` also marks
`root.SeenProbes[key]` after popping. This caused root's own probes to be
marked before execution and then skipped by the scheduler ‚Äî probes were
generated but never ran.

### Changes

**`behavior/expand.go`**
- `pushProbe` now only marks `SeenProbes` immediately for non-root entities
- Root's `SeenProbes` remains exclusively owned by the scheduler in `runQueuedProbes`

---

## Fix 2 ‚Äî Auth boundary detection for redirect responses

### Problem
`detectEndpointAuthGate` bailed out if `baseline != 200`. Endpoints that
return 3xx (e.g. login form POST returning 303 on success) never triggered
auth boundary detection ‚Äî the identity comparison loop never ran.

### Changes

**`behavior/run.go`**
- Replaced `baseline != 200` check with `baseline >= 200 && baseline < 400`
- Any 2xx or 3xx baseline response now counts as "baseline succeeded"

---

## Fix 3 ‚Äî Sub-probe propagation infinite loop

### Problem
When a probe targeted the root URL, `Expand(target)` was called where
`target == root`. The propagation block then ran unconditionally, popping
from `root.ProbeQueue` and pushing back probes that passed the `SeenProbes`
check ‚Äî creating an infinite loop. The outer `Run()` loop kept cycling
because `after > before` was always true.

### Changes

**`behavior/run.go`**
- Added `target != root` guard to the propagation block
- When target is root, `Expand` already pushes directly to root's queue ‚Äî
  no propagation needed

---

## Fix 4 ‚Äî Discovery probe flooding

### Problem
`expandDiscovery` was called on every `Expand(target)` invocation because
the `meaningful` check (requiring `LikelyObjectAccess`, `Enumerable` etc.)
was never satisfied on fresh entities. This caused repeated discovery sweeps
generating the same probes on every probe execution cycle.

### Changes

**`knowledge/entity.go`**
- Added `DiscoveryProbed bool` to `EntityState`

**`behavior/expand.go`**
- `expandDiscovery` returns immediately if `ent.State.DiscoveryProbed` is true
- Sets `DiscoveryProbed = true` before pushing any discovery probes
- Each entity now gets exactly one discovery sweep

---

## Fix 5 ‚Äî Credentialless token issuance detection

### Problem
The engine had no way to detect when a server issues authentication tokens
(JWTs, session cookies) to requests that sent no credentials. This is a real
security finding ‚Äî a login endpoint that issues valid tokens to anonymous
POSTs is misconfigured regardless of what those tokens grant access to.

### Changes

**`knowledge/signals.go`**
- Added `SigCredentiallessTokenIssuance` signal
- Added `"CredentiallessTokenIssuance"` to `Signal.String()`

**`behavior/analyze.go`**
- Added `analyzeCredentiallessIssuance(ent *knowledge.Entity)`
- Fires when: entity is stateful (`SigStateChanging`, `SessionIssued`, or
  `AuthLikely`) AND any identity with `SentCreds=false` has
  `IssuedByServer=true` and `Rejected=false`
- Wired into `runQueuedProbes` after `analyzeMethods`

**`report/derive.go`**
- Added finding: "Server issues tokens/sessions without credentials ‚Äî
  credentialless authentication bypass possible"
- Added next steps: test whether issued tokens grant authenticated access,
  attempt token reuse across sessions

---

## Fix 6 ‚Äî Method detection for redirect responses

### Problem
`methodAllowed` returned `false` for all 3xx status codes. Endpoints that
respond with redirects (e.g. POST ‚Üí 303, common in login form handling) had
no methods recorded despite responding correctly to method probes.

### Changes

**`behavior/run.go`**
- Added 301, 302, 303, 304, 307, 308 to the `methodAllowed` allow-list
- Redirect responses now correctly register the probed method on the entity

---

## Fix 7 ‚Äî Debug flag not reaching behavior engine

### Problem
`behavior.New()` had `debug` hardcoded to `false`. The `--debug` flag from
the CLI was parsed into `req.Flags.Debug` but never passed into the engine,
so behavior-level debug output never appeared regardless of the flag.

Additionally `ReconHandler.Translate` was unconditionally injecting `--debug`
into every recon run, and `http`/`web` profile words were not consumed by
`isProfile` ‚Äî they leaked into the args before `--debug`, causing Go's flag
parser to stop at the unknown word and never register the flag.

### Changes

**`behavior/engine.go`**
- `New(k, i, debug bool)` ‚Äî added `debug` parameter, wired to `e.debug`

**`internal/recon/http/engine.go`**, **`internal/recon/api/engine.go`**
- Updated `behavior.New` call to pass `req.Flags.Debug`

**`intent/recon.go`**
- Removed unconditional `--debug` injection from `ReconHandler.Translate`
- Replaced `isProfile` call with `isReconProfile` ‚Äî new function scoped to
  recon that recognizes `http`, `web`, and `api` without polluting the shared
  `isProfile` used by fetch/send/upload handlers

**`intent/words.go`**
- Added `isReconProfile(word string) (string, bool)` ‚Äî recognizes `http`,
  `web`, `api` as valid recon profiles and consumes them from the arg list
  before flag parsing runs
