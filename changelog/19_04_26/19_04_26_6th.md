# cwrap recon — session resilience & token handling
```19/04/26```

## Overview

Five fixes this session: stale session auto-recovery, role boundary detection
with expired tokens, TLS skip verification, redirect destination probing, and
live session cookie propagation across probe executions.

---

## Fix 1 — Stale session auto-recovery

### Problem
When a session saved from a previous run contained expired tokens, the
baseline request returned 401 and the engine continued probing with dead
credentials. The user had to manually delete
`~/.config/cwrap/sessions/<host>.json` to recover.

### Changes

**`behavior/run.go`** — added stale session detection block in `Run()` between
the baseline request and identity extraction:
- If baseline returns 401 AND session cookies were injected (`ent.SessionUsed`)
  → stale session detected
- Clears in-memory session cookies (`ent.SessionCookies`, `e.sessionCookies`, `ent.SessionUsed`)
- Rebuilds `baseReq` from the original request
- Re-derives identities from the clean request
- Retries the baseline request without stale credentials
- If the retry also returns 401 — proceeds normally as unauthenticated

Debug output: `== Stale session detected (401 on baseline) — retrying without session cookies ==`

## Fix 2 — Role boundary detection with expired tokens

### Problem
`detectRoleBoundary` only checked for status 403. When a valid but expired
token was used, the server returned 401 instead of 403 — the `RoleBoundary`
signal never fired even though a credentialed identity was clearly denied.

### Changes

**`behavior/run.go`** — `detectRoleBoundary`:
- Now checks `status != 401 && status != 403` to skip
- Both 401 and 403 from an identity with `SentCreds=true` trigger `SigRoleBoundary`

## Fix 3 — TLS certificate verification disabled

### Problem
Go's default `http.Client` refuses to connect to HTTPS targets with
self-signed certificates. Security testing targets — HTB machines, lab
environments, internal apps — almost always use self-signed certs.

### Changes

**`internal/transport/client.go`**
- Added custom `http.Transport` with `tls.Config{InsecureSkipVerify: true}`
- Equivalent to `curl -k` — skips certificate validation, still uses TLS
- Applied unconditionally
- Added `crypto/tls` import

## Fix 4 — Redirect destination probing

### Problem
3xx responses were recorded but their `Location` destination was never
queued for probing. Post-auth redirects like `POST /login/submit → 303 →
/home` meant the destination endpoint was never analyzed.

### Changes

**`internal/recon/http/learn.go`** and **`internal/recon/api/learn.go`**
- After `httpintel.Learn`, extract `Location` header on any 3xx response
- Normalize the location URL against the base URL
- Add a graph edge and push a GET probe via `e.k.PushProbe`
- Priority 80 — higher than param discovery, lower than identity probes

**`internal/recon/api/normalize_link.go`**
- Added `normalizeLink` to the api engine — same implementation as http engine

**`internal/recon/knowledge/constants.go`**
- Added `ReasonRedirect = "redirect"`

## Fix 5 — Live session cookie propagation

### Problem
The baseline identity's `Apply` function closed over `baseReq` at derivation
time. When `captureSession` captured fresh tokens mid-run (e.g. from a login
endpoint), those tokens were saved to the session file but never used by
subsequent probes in the same run — baseline kept sending the original
cookies.

### Changes

**`behavior/engine.go`**
- Added `sessionCookies map[string]string` field — live cookie store updated
  throughout the run

**`behavior/identity.go`** — `deriveIdentities` is now a method on `*Engine`:
- Baseline `Apply` reads from `e.sessionCookies` dynamically on every probe
- Merges session cookies with user-supplied cookies (user takes priority)
- If `e.sessionCookies` is empty, returns request unchanged

**`behavior/session.go`** — `captureSession`:
- Added `e.sessionCookies[c.Name] = c.Value` alongside existing
  `ent.SessionCookies` update — keeps engine cookie store in sync

**`behavior/run.go`**:
- Removed old static cookie injection into `baseReq`
- Populates `e.sessionCookies` from persisted store after loading session
- Stale session retry clears both `ent.SessionCookies` and `e.sessionCookies`
