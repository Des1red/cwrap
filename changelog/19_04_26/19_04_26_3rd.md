# cwrap recon — engine cleanup & probe dedup centralization
```19/04/26```
## Overview

Cleanup session across the http engine, api engine, and knowledge layer.
No new features — dead code removed, dedup centralized, classification gaps
filled, and a premature signal tag fixed.

---

## Fix 1 — Premature `SigAuthBoundary` in httpintel

### Problem
`httpintel.Learn` was tagging `SigAuthBoundary` on any single 401 or 403
response. This bypassed the mixed-access requirement enforced by
`analyzeAuthBoundary` in the behavior layer — every denied endpoint got
flagged regardless of whether any identity ever succeeded.

### Changes

**`internal/recon/httpintel/learn.go`**
- Removed `ent.Tag(knowledge.SigAuthBoundary)` from the 401/403 block
- `AuthLikely = true` preserved — appropriate single-response signal
- `SigAuthBoundary` is now exclusively owned by `analyzeAuthBoundary` and
  `analyzeMethods` in the behavior layer

---

## Fix 2 — Probe dedup centralized into `knowledge.PushProbe`

### Problem
Three separate call sites each had their own partial dedup logic:
- `behavior/expand.go` (`pushProbe`) — checked `SeenProbes` for non-root entities
- `http/js.go` (`handleJSEndpoints`) — bypassed `SeenProbes` entirely, used direct `ProbeQueue.Push`
- `behavior/run.go` (propagation block) — checked `root.SeenProbes` manually before pushing

This meant JS-discovered endpoints could enter the queue multiple times, and
any future caller adding probes would need to re-implement the same logic.

### Changes

**`internal/recon/knowledge/probe.go`**
- Added `Knowledge.PushProbe(target *Entity, p Probe)` — canonical dedup entry point
- Checks `root.SeenProbes` first (global gate), then `target.SeenProbes` for non-root entities
- Marks `target.SeenProbes[key] = true` before pushing
- `ProbeQueue.Push` renamed to `push` (lowercase — internal only)
- `push` simplified to plain append — dedup is now `PushProbe`'s responsibility

**`behavior/expand.go`**
- `pushProbe` is now a thin wrapper: `e.k.PushProbe(ent, p)`

**`http/js.go`**
- `handleJSEndpoints` now uses `e.k.PushProbe` — manual root check and direct queue push removed

**`behavior/run.go`**
- Propagation block now uses `e.k.PushProbe(root, p)` — manual `SeenProbes` check removed

### Problem

## Fix 5 — `walkJSON` missing param classification

### Problem
`walkJSON` in the api engine called `ent.AddParam` and `e.k.AddParam` for
JSON-discovered keys but never called `paramintel.ClassifyParam`. This meant
JSON params never got `IDLike`, `TokenLike`, or `DebugLike` set, and signals
like `SigIDLikeParam` never fired for API responses.

### Changes

**`internal/recon/api/json.go`**
- Added `paramintel.ClassifyParam(ent, ent.Params[k])` after each `AddParam`
  call in `walkJSON`
- Added `paramintel` import

---

## Fix 6 — Nil panic risk in api interpreter

### Problem
`api/interpreter.go` `ClassifyParam` passed `ent.Params[name]` directly to
`paramintel.ClassifyParam` without checking if the param existed. If `name`
was not in `ent.Params`, a nil `*ParamIntel` was passed and
`paramintel.ClassifyParam` would panic on `p.Name` dereference.

### Changes

**`internal/recon/api/interpreter.go`**
- Added `p, ok := ent.Params[name]` guard — returns early if param not found
- Mirrors the existing guard pattern in `http/interpreter.go`
