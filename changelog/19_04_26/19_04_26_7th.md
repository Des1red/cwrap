# cwrap recon — JWT intelligence & identity depth
`2026-04-19`

## Overview

JWT claims are now decoded and stored on identities during probing.
Role, user ID, and expiry are visible directly in the report without
manual decoding. The identity preservation ratchet was extended to
cover the new fields.

---

## Fix 1 — JWT claim extraction

### Problem
`extractIdentity` extracted only the `role` claim from JWT tokens found
in cookies and bearer headers. `user_id`, `exp`, and `jti` were parsed
and discarded. The report showed `role=member` but not which user or
when the token expires.

### Changes

**`behavior/jwt.go`**
- Added `extractJWTIntel(id *knowledge.Identity, claims jwtClaims)` —
  extracts `role`, `user_id`, `exp` (formatted as `2006-01-02 15:04`
  UTC), and `jti` from parsed claims
- `extractRoleFromClaims` kept for backward compatibility

**`behavior/extractidentity.go`**
- All three JWT parse sites (`sent cookies`, `issued cookies`,
  `bearer header`) now call `extractJWTIntel` instead of
  `extractRoleFromClaims`
- Preservation ratchet extended — `UserID`, `Expiry`, `TokenJTI` are
  now preserved across probe executions alongside `Role`, `SentCreds`,
  `Effective`

**`knowledge/identity.go`**
- Added `UserID string`, `Expiry string`, `TokenJTI string` fields to
  `Identity`

---

## Fix 2 — Elevated role expanded

### Problem
`isElevatedRole` did not include `owner` as an elevated role. Tokens
with `role=owner` were classified as `IdentityUser` instead of
`IdentityElevated`.

### Changes

**`behavior/extractidentity.go`**
- Added `strings.Contains(r, "owner")` to `isElevatedRole`

---

## Fix 3 — Report identity line enriched

### Problem
The report identity line showed kind, creds, rejected, issued-token,
effective, role, scheme, csrf — but not user ID or expiry. A reviewer
had to manually decode the JWT from the session section to see these.

### Changes

**`report/full.go`**
- Identity line now includes `uid=X` and `exp=YYYY-MM-DD HH:MM` when
  present
- Example:
  `anonymous: [IdentityElevated issued-token role=owner uid=42 exp=2026-04-20 08:45]`

---

## What this enables

Two report reads now tell a complete attack chain without manual work:

- `/login/admin` report shows:
  `anonymous: [IdentityElevated issued-token role=owner uid=42 exp=2026-04-20 08:45]`
  — free owner token, valid for hours, issued to unauthenticated identity

- `/admin` report shows:
  `RoleBoundary` signal + `role=owner` required

- Conclusion: take the token from `/login/admin` session section,
  use it against `/admin` — no JWT decoding required
