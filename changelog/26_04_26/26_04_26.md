# Changelog

`2026-04-25` `v2.6.1`

### Scan — Stage 3: Subdomain Enumeration
- Added `stageThree` — enumerates subdomains of the apex host extracted from the target URL
- Wildcard DNS detection: probes two random subdomains before scanning; filters matching responses by content hash or size band (same logic as soft 404)
- `apexHost` strips leading subdomain components and handles common ccTLD two-parters (`co.uk`, `com.au`, etc.)
- Stage 3 skips automatically on localhost/IP targets unless `--domain` is explicitly provided; prints a clear skip message with the reason
- Distinct baseline messaging: "Wildcard DNS detected" vs "Soft 404 detected"

### Scan — Flags: `--dir` and `--domain`
- Directory wordlist is now `--dir <path>` instead of a positional argument
- Subdomain wordlist is `--domain <path>`
- Both fall back to bundled defaults if not provided
- Positional wordlist argument removed from `scan` — `intent.Parse` no longer consumes a third positional for scan
- `ScanHandler.Translate` recognises profile words (`browser`, `firefox`, `chrome`, `api`, `curl`) and wires them into `f.Profile`
- `mergeResults` made variadic to accommodate three stages without signature changes

### Scan — Header stack via `httpcore`
- `newClient` now takes `model.Request` and applies the full cwrap header stack on every probe via `authTransport`
- `authTransport.RoundTrip` calls `httpcore.BuildHeaders` for profile/bearer/custom headers, then adds cookies on top
- `--debug` prints headers once on the first probe rather than on every request

### Recon — Profile and engine separation
- `ReconMode` added to `model.Request` — engine selection (`http`/`api`) is now independent from the request header profile
- `selectEngine` reads `req.ReconMode` instead of `req.Flags.Profile`
- `ReconHandler` separates `reconMode` (engine) and `profile` (headers) — both can be set independently
- `isReconProfile` and `isProfile` kept distinct: recon modes do not imply header profiles by default
- `api.Engine.Run` sets `req.Flags.Profile = "api"` as its own default if no explicit profile was provided — every `transport.Do` call inside the behavior loop sends `Accept: application/json`
- `transport.Do` prints outgoing request headers in debug mode via `dumpRequest`

### Exploit — Profile support
- `ExploitHandler` now recognises profile words and sets `f.Profile`
- `DoGETWithCookies` takes `model.Flags` and applies `httpcore.BuildHeaders` before injecting the session jar — profile headers apply to every replay request
- Session jar cookies merged after flag cookies; jar wins on name collision
- `NewHTTPClient` adds TLS skip to match the scan client
- `expandFlags` removed — all expand functions now take `model.Flags` directly

### Model
- `model.Request.ReconMode string` added
- `model.Flags.DirWordlist string` and `model.Flags.DomainWordlist string` added

### Wordlists
- `wordlists/small-directory-list-20k.txt` — ~19,500 entries: core dirs, API sub-routes, admin panels, Spring Boot actuator paths, CMS-specific paths, cloud metadata, sensitive file targets, dev artifacts, dot files, monitoring stack paths
- `wordlists/small-subdomain-list-20k.txt` — ~19,200 entries: core infra, environment variants, region × service combos, devops tool names, corp/internal tools, product/SaaS names, numbered variants, industry verticals

## Earlier

### Scan — Stages 1 and 2
- Stage 1: parallel directory discovery with soft 404 baseline fingerprinting (hash and size-band modes)
- Stage 2: subdirectory expansion — every 200 from Stage 1 is re-scanned with the full wordlist in parallel
- 20 concurrent workers, progress counter, results saved to `<host>_scan.txt`

### Recon — Core engine
- HTTP and API engines with shared behavior loop
- Identity derivation: session, anonymous, fake-admin (role confusion headers)
- Dynamic identity registration from JWT claims
- Path-level ID segment detection and mutation probing
- Session persistence to `~/.config/cwrap/sessions/<host>.json`
- Stale session detection and automatic retry
- Signal detection: AuthBoundary, RoleBoundary, ObjectOwnership, PossibleIDOR, CredentiallessTokenIssuance, AdminSurface, PublicAccess, StateChanging, SensitiveKeyword
- Full report serialisation to `reports/<host>_<timestamp>.report`

### Exploit — Core engine
- Stage 1: ownership, IDOR, and credentialless probes against report findings
- Stage 2: credentialless token reuse expansion, IDOR object enumeration, ownership bypass pivot
- Identity vault replay using frozen cookie snapshots from recon

### httpcore
- `BuildHeaders` — merges profile headers, content profile headers, user-specified headers, bearer token
- Profiles: `firefox`, `chrome`, `api`, `curl`
- Content profiles: `json`, `form`, `xml`
- Used by `transport.Build` (recon), `authTransport` (scan), and `DoGETWithCookies` (exploit)

### Intent / flag parsing
- Semantic word parser: profile words, recon mode words, content words, boolean words, cookie tokens, auth tokens
- `FetchHandler`, `SendHandler`, `UploadHandler`, `ReconHandler`, `ScanHandler`, `ExploitHandler`
- All handlers wire into the same flag parsing pipeline; raw curl-compatible flags pass through unchanged
