## [FIX] IDOR surfaces = 0 despite correct ownership boundary detection

**File:** `behavior/analyze.go` — `analyzeIDOR`

### Problem

`analyzeIDOR` reported 0 IDOR surfaces even when `analyzeOwnership` had correctly identified mixed credentialed access across path ID values. The scan output showed:

```
Ownership boundary:  yes
IDOR surfaces:       0   ← bug
```

Despite `storeResponse` debug confirming correct per-entity accumulation:

```
/api/notes/1: session=200, member-uid-1=200, member-uid-2=403, owner-uid-3=200
/api/notes/2: session=403, member-uid-1=403, member-uid-2=200, owner-uid-3=200
```

### Reason

`noCredDenied` was only ever set by `anyNoCredDenied()`, which exclusively checks for *unauthenticated* identity denials (`id.SentCreds == false`). In the ownership case, all probed identities are credentialed — no anonymous identity is ever denied — so `anyNoCredDenied` always returned `false`, blocking every IDOR signal path regardless of what `analyzeOwnership` had already confirmed.

The two existing signal paths both hard-gated on `noCredDenied`:

```go
// WEAK signal — never fired
if len(rawFPs) >= 2 && noCredDenied && p.IDLike { ... }

// STRONG signal — never fired
if structDiff {
    if noCredDenied {          // ← always false in ownership case
        p.PossibleIDOR = true
    }
}
```

### Fix

**1.** Added a second `noCredDenied = true` gate inside the `byVal` loop that fires when some credentialed identities are allowed and others are denied on the same value:

```go
// ownership IDOR: some cred identities allowed, others denied on this same value
if anyCredStatus(ent, byIDStatus, 200) &&
    (anyCredStatus(ent, byIDStatus, 403) || anyCredStatus(ent, byIDStatus, 401)) {
    noCredDenied = true
}
```

**2.** Added an ownership IDOR emit block at the bottom of the param loop, guarded by `p.OwnershipBoundary && noCredDenied`, so a structural diff is not required when ownership is already proven by `analyzeOwnership`:

```go
// --- OWNERSHIP IDOR (no structural diff needed, ownership already proven) ---
if p.OwnershipBoundary && noCredDenied {
    p.PossibleIDOR = true
    ent.Tag(knowledge.SigPossibleIDOR)
}
```

**3.** Call order in `runQueuedProbes` verified — `analyzeOwnership` always runs before `analyzeIDOR` in both the per-probe path and the path ID source entity path, so `p.OwnershipBoundary` is always set before the IDOR check reads it. No race condition.

### Result

```
Ownership boundary:  yes
IDOR surfaces:       1   ← fixed

High Risk Findings:
 • Horizontal privilege escalation suspected: http://localhost:9001/api/notes/1 (param: note_id)
```

Confirmed by exploit engine — member-uid-1 successfully accessed Alice's private note (`owner_id=1`) returning full object content. Ownership correctly enforced in both directions for `/api/notes/2`.

## [FIX] Session terminator guard — live session killed by identity/method probes

**Files:** `behavior/expand.go` — `expandIdentity`, `expandMethods`

### Problem

cwrap probed `GET /api/logout` with the live session cookie during both the identity sweep and the method sweep, causing the server to invalidate the session. All subsequent authenticated probes failed.

### Reason

`expandMethods` sweeps all HTTP verbs (including POST and DELETE) against every discovered endpoint. `expandIdentity` probes every auth-signalled endpoint with the live session. Neither had any awareness of session-terminating URL patterns.

`isDestructiveToken` already existed in `html.go` for form submission safety but is intentionally broader (covers delete/remove/disable etc.) and lives in the wrong package for this use case.

### Fix

Added `isSessionTerminator(u string) bool` to `behavior/expand.go` covering logout/signout/logoff/session-destroy patterns. Added early-return guards to both functions:

```go
// expandIdentity — skip entirely
if isSessionTerminator(ent.URL) {
    return
}

// expandMethods — mark as done, skip sweep
if isSessionTerminator(ent.URL) {
    ent.State.MethodProbed = true
    return
}
```

The edge is still recorded in the graph so the discovery tree shows the relationship. Only probes are suppressed.

## [FIX] SPA shell reflection — phantom JS entity explosion

**Files:** `internal/recon/http/html.go`, `internal/recon/jsintel/jsintel.go`, `internal/recon/knowledge/knowledge.go`

### Problem

Recon on a large SPA target produced 130 entities and 542 JS leaks instead of ~40 and 0. JS files under `/js/` were being re-fetched and re-analyzed under every API path prefix that returned the SPA HTML shell.

### Reason

The server returned the same `index.html` shell for every route including API routes. The shell contains `<script src="js/app.js">` (relative). When cwrap fetched `/api/logout` and got the shell back, `extractHTML` resolved `js/app.js` relative to `/api/logout` → `/api/js/app.js`, queued a fetch probe, created a new entity, fetched it, got the shell again, and the cycle repeated for every SPA route discovered. Each copy of the JS tree ran through jsintel and emitted duplicate findings.

### Fix

**1. `knowledge.Knowledge`** — added `KnownJSSuffixes sync.Map` (zero-value ready, no constructor needed) with two methods:

```go
func (k *Knowledge) RegisterJSSuffix(suffix string) bool {
    _, loaded := k.KnownJSSuffixes.LoadOrStore(suffix, struct{}{})
    return !loaded // true = first registration, false = already existed
}

func (k *Knowledge) HasJSSuffix(suffix string) bool {
    _, ok := k.KnownJSSuffixes.Load(suffix)
    return ok
}
```

`LoadOrStore` makes the check-and-register atomic, eliminating the race between goroutines processing different SPA routes concurrently.

**2. `jsintel`** — added two helpers:

```go
func JSPathSuffix(rawURL string) string       // extracts /js/... from a resolved URL path
func IsPhantomJSURL(k *knowledge.Knowledge, resolvedURL string) bool
```

**3. `html.go` `case "script"`** — before queuing a JS fetch probe, check for phantom and atomically register:

```go
if !jsintel.IsPhantomJSURL(e.k, link) {
    if suffix := jsintel.JSPathSuffix(link); suffix != "" {
        if !e.k.RegisterJSSuffix(suffix) {
            break // lost race — another goroutine registered first
        }
    }
    e.k.PushProbe(...) // only reached on first registration
}
```

`AddEdge` is still called for every phantom (graph relationship preserved), but no probe is queued so no entity is ever created for the duplicate.

### Result

```
Before: Entities=130  JS leaks=542
After:  Entities=40   JS leaks=0
```

## [FIX] SVG/XML namespace noise in JS host_url findings

**File:** `internal/recon/jsintel/jsintel.go`

### Problem

`reURL` matched XML namespace declarations in inline SVG (`http://www.w3.org/2000/svg`) and reported them as `host_url` leaks.

### Fix

Added `isNoiseURL(u string) bool` filtering known semantic-noise domains (W3C, XML schemas, Dublin Core, OGP, schema.org, etc.) and applied it in the `host_url` accumulation loop before counting or recording any leak. The loop was also corrected to only count non-noise hits — the previous implementation counted all regex matches including noise before applying the cap.

### Result

```
Before: JS leaks=2  (both false positives — SVG namespace URLs)
After:  JS leaks=0
```
