# cwrap scan — Changelog

`2026-04-27` `v2.6.3`

---

## Remove `plainClient` from scan pipeline

**Issue**
`stageThree` received a `plainClient` (no auth headers) while all other stages used `scanClient` (full auth stack). Subdomain probes were sent without authentication, cookies, or custom headers.

**Reason**
`plainClient` was introduced as a "baseline only" client but was never justified for subdomain enumeration. There is no reason subdomain probes should bypass the configured auth stack.

**Fix**
Removed `newPlainClient()` and its instantiation in `Run`. `stageThree` now receives `scanClient` along with all other stages.

---

## Introduce `doDirScan` / `doSubdomainScan` flags

**Issue**
Running only a subdomain scan was not possible. Providing `--domain` without `--dir` would still attempt directory scanning and error if no directory wordlist was found.

**Reason**
`validate()` was a single gate that required a directory wordlist unconditionally. There was no concept of scan mode — the tool always tried to do everything.

**Fix**
Replaced `validate()` with two symmetric validators: `validateDirScan` and `validateSubdomainScan`. Each attempts to resolve its wordlist (explicit flag → bundled default), mutates `req` on success, and returns an error on failure. `Run` evaluates both at startup and sets `doDirScan` / `doSubdomainScan` booleans accordingly. Stages only execute when their flag is true.

---

## Remove `resolveSubdomainFile`

**Issue**
`resolveSubdomainFile` duplicated the wordlist resolution logic that now lives in `validateSubdomainScan`, and was called redundantly at the `stageThree` call site.

**Reason**
After introducing `validateSubdomainScan`, `req.SubdomainFile` is already resolved and populated before `Run` reaches the subdomain block. A second resolution step was redundant.

**Fix**
Deleted `resolveSubdomainFile`. `stageThree` now receives `req.SubdomainFile` directly.

---

## Stage 2 — suppress per-directory "no findings" noise

**Issue**
When subdirectory expansion found nothing, a "no findings under `<url>`" line was printed for every expanded directory individually, producing cluttered output proportional to the number of dirs.

**Reason**
The check was inside the goroutine, firing per-directory before results were merged.

**Fix**
Moved the check after `wg.Wait()`. A single `no findings` line is printed once against the merged result.

---

## Stage 2 — gate "expanding" line behind `--debug`

**Issue**
`expanding <url>` was printed for every directory unconditionally, adding noise to normal output.

**Reason**
Expansion progress is a diagnostic detail, not a user-facing result.

**Fix**
`stageTwo` now accepts a `debug bool` parameter. The `expanding` line is gated behind it. Call site passes `req.Flags.Debug`.

---

## Subdomain scan — always block localhost and IP targets

**Issue**
Passing `--domain` with an explicit wordlist bypassed the localhost/IP guard, allowing subdomain enumeration against `localhost`, `127.0.0.1`, or `::1`.

**Reason**
The original guard short-circuited when an explicit file was provided, treating it as a user override. Subdomain enumeration against loopback or IP addresses is never meaningful regardless of intent.

**Fix**
Removed the early return for explicit `subdomainFile`. Host type is now checked unconditionally. The `--domain` flag controls the wordlist but cannot override the target type guard. Error message updated to drop the "use --domain to force" suggestion since forcing is no longer possible.
