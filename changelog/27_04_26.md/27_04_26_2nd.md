## [FIX] IDOR surfaces = 0 despite correct ownership boundary detection

**File:** `behavior/analyze.go` — `analyzeIDOR`

### Problem

`analyzeIDOR` reported 0 IDOR surfaces even when `analyzeOwnership` had correctly identified mixed credentialed access across path ID values. The scan output showed:

```
Ownership boundary:  yes
IDOR surfaces:       0   ← bug
```

Despite `storeResponse` debug confirming correct per-entity accumulation:

```
/api/notes/1: session=200, member-uid-1=200, member-uid-2=403, owner-uid-3=200
/api/notes/2: session=403, member-uid-1=403, member-uid-2=200, owner-uid-3=200
```

### Reason

`noCredDenied` was only ever set by `anyNoCredDenied()`, which exclusively checks for *unauthenticated* identity denials (`id.SentCreds == false`). In the ownership case, all probed identities are credentialed — no anonymous identity is ever denied — so `anyNoCredDenied` always returned `false`, blocking every IDOR signal path regardless of what `analyzeOwnership` had already confirmed.

The two existing signal paths both hard-gated on `noCredDenied`:

```go
// WEAK signal — never fired
if len(rawFPs) >= 2 && noCredDenied && p.IDLike { ... }

// STRONG signal — never fired
if structDiff {
    if noCredDenied {          // ← always false in ownership case
        p.PossibleIDOR = true
    }
}
```

### Fix

**1.** Added a second `noCredDenied = true` gate inside the `byVal` loop that fires when some credentialed identities are allowed and others are denied on the same value:

```go
// ownership IDOR: some cred identities allowed, others denied on this same value
if anyCredStatus(ent, byIDStatus, 200) &&
    (anyCredStatus(ent, byIDStatus, 403) || anyCredStatus(ent, byIDStatus, 401)) {
    noCredDenied = true
}
```

**2.** Added an ownership IDOR emit block at the bottom of the param loop, guarded by `p.OwnershipBoundary && noCredDenied`, so a structural diff is not required when ownership is already proven by `analyzeOwnership`:

```go
// --- OWNERSHIP IDOR (no structural diff needed, ownership already proven) ---
if p.OwnershipBoundary && noCredDenied {
    p.PossibleIDOR = true
    ent.Tag(knowledge.SigPossibleIDOR)
}
```

**3.** Call order in `runQueuedProbes` verified — `analyzeOwnership` always runs before `analyzeIDOR` in both the per-probe path and the path ID source entity path, so `p.OwnershipBoundary` is always set before the IDOR check reads it. No race condition.

### Result

```
Ownership boundary:  yes
IDOR surfaces:       1   ← fixed

High Risk Findings:
 • Horizontal privilege escalation suspected: http://localhost:9001/api/notes/1 (param: note_id)
```

Confirmed by exploit engine — member-uid-1 successfully accessed Alice's private note (`owner_id=1`) returning full object content. Ownership correctly enforced in both directions for `/api/notes/2`.
