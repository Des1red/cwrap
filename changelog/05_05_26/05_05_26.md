# Changelog 2

```05-05-26``` ```v2.7.4```

## 1. Contact information not captured

**Reason:** `mailto:` and `tel:` links in HTML contain actionable recon data (email addresses, phone numbers) that are useful for social engineering and account enumeration during exploitation. These were being silently dropped during link normalization.

**Problem:** `normalizeLink` rejected `mailto:` and `tel:` scheme URLs before any extraction could occur, so contact info embedded in anchor tags was never recorded.

**Fix:** Added `Emails map[string]bool` and `Phones map[string]bool` to `Knowledge` with `AddEmail()` and `AddPhone()` methods. Extraction runs in `extractHTML` before `normalizeLink` is called, parsing `href="mailto:..."` and `href="tel:..."` values directly. Added `writeContactInfo()` section to the full report.

---

## 2. CSRF header not injected on mutating requests


**Reason:** Laravel and similar frameworks require mutating requests (POST/PUT/PATCH/DELETE) to include `X-XSRF-TOKEN` header set to the URL-decoded value of the `XSRF-TOKEN` cookie. Without it, the server returns 419.

**Problem:** cwrap never injected a CSRF header on mutating probes. POST probes to Pterodactyl were consistently returning 419, meaning no real mutation surface was being tested.

**Fix:** Added `injectCSRFHeader` helper in `behavior/run.go`. Called in `runQueuedProbes` after `id.Apply(reqID)` for POST/PUT/PATCH/DELETE methods. Reads the CSRF token already tracked on `target.Identities[id.Name]` (populated by `extractIdentity` on every previous response) and injects the appropriate header (`X-XSRF-TOKEN` for XSRF cookies, `X-CSRF-TOKEN` otherwise) with the URL-decoded value. Falls back to scanning `e.sessionCookies` for first-probe scenarios where the identity has not yet been observed on the target entity.

## 3. CSRF header name inconsistent across probes

**Reason:** `extractIdentity` preserves `CSRFToken` from previous responses via the preserve block, but did not preserve `CSRFCookieName`. When a subsequent response did not rotate the CSRF cookie, `CSRFCookieName` was empty on the new identity object while `CSRFToken` was carried over from prev.

**Problem:** `csrfHeaderName("")` falls through to `X-CSRF-TOKEN` since the empty string does not contain "XSRF". This caused some probes to send `X-Csrf-Token` and others to correctly send `X-Xsrf-Token` depending on whether the most recent response happened to rotate the XSRF-TOKEN cookie.

**Fix:** Added `id.CSRFCookieName = prev.CSRFCookieName` to the preserve block in `extractIdentity`, alongside the existing `CSRFToken` preservation. All probes now consistently send `X-Xsrf-Token` for XSRF-based targets.

---

## 4. Synthetic identities suppressed on security-critical probes after auth boundary confirmation

**Reason:** Once `authBoundaryConfirmed` is set, all probes skip synthetic identities (anonymous, corrupted token, fake-admin). The intent was to avoid flooding param discovery probes with anonymous requests that would always get 401. However, the guard was applied globally to every probe regardless of reason.

**Problem:** Method probes (`expandMethods`) and identity probes (`expandIdentity`) also had synthetic identities suppressed. This meant anonymous and fake-admin were never tested against sub-endpoints after the boundary confirmed. `detectAuthBoundary`, `detectPublicAccess`, and `detectRoleBoundary` could never fire on any sub-endpoint because `identityStatuses` only ever contained the session identity. No public endpoints were discovered even when they existed.

**Fix:** Replaced the blanket guard with a reason-scoped check. Synthetic identities are now suppressed only for `ReasonParamDiscovery`, `ReasonIDAdjacency`, `ReasonIDEnum`, and `ReasonLinkProbe` — probes where anonymous coverage adds noise with no signal. `ReasonMethodProbe` and `ReasonIdentityProbe` now run all identities regardless of `authBoundaryConfirmed`, restoring full security coverage on every discovered endpoint. Verified: 3 public endpoints detected on Pterodactyl that were previously invisible.
