# cwrap recon — accuracy improvements

## Overview

Four targeted fixes to the recon engine's active probing and reasoning layer.
All changes are confined to `internal/recon/behavior/` and `internal/recon/knowledge/`.

---

## Fix 1 — Path segment ID detection

### Problem
The engine only handled query-string params (`?id=123`). REST-style path IDs
like `/users/123` or `/posts/456/comments` were completely invisible — no probes
were generated, no IDOR analysis was possible against them.

### Changes

**`knowledge/probe.go`**
- Added `PathParams map[string]string` — path segment param name → substituted value
- Added `PathParamBase map[string]string` — original segment value, used for base seeding in analyzers
- Updated `Probe.Key()` to include path params with `pp:` prefix for correct deduplication

**`knowledge/constants.go`**
- Added `ReasonPathIDProbe = "path-id-probe"`

**`knowledge/entity.go`**
- Added `MethodProbed bool` to `EntityState` (used by fix 3)

**`behavior/expand.go`**
- Added `looksLikePathID(s string) bool` — detects numeric segments and UUIDs
- Added `replacePathSegment(u *url.URL, idx int, val string) string` — rebuilds URL with substituted segment
- Added `expandPathIDs(ent *knowledge.Entity)` — detects ID segments, derives contextual param names (e.g. `/users/123` → `user_id`), generates mutation probes with adaptive depth based on Interest score
- Wired `expandPathIDs` into `Expand()` after `expandMethods`

**`behavior/store.go`**
- Unified the two duplicate storage loops (`AddQuery` and `PathParams`) into a single loop over a `[]paramEntry` slice
- Path params use `PathParamBase` for base seeding instead of `extractCurrentValue`

**`behavior/run.go`**
- Added path param registration block in `runQueuedProbes` (registers params on target entity with `ParamPath` source)
- Added sub-probe propagation: after `Expand(target)`, any probes pushed to a non-root entity's queue are moved to root's queue so they actually execute. Previously these were silently dropped.

## Fix 2 — Auth boundary detection

### Problem
`detectEndpointAuthGate` only compared `baseline` vs `anonymous`. This meant:
- `corrupted-token` and `fake-admin` identities were ignored entirely
- Sites that return 200 with an empty body instead of a hard 401/403 never triggered the flag
- Once the flag was unset it stayed unset regardless of accumulating evidence

### Changes

**`behavior/engine.go`**
- Added `authConfidence int` field to `Engine`

**`behavior/run.go`**
- Updated `detectEndpointAuthGate` call site to pass `probeFP` as second argument
- Rewrote `detectEndpointAuthGate(identityStatuses map[string]int, probeFP map[string]string)`:
  - Loops all non-baseline identities instead of only `anonymous`
  - Hard signal: any non-baseline identity gets 401/403 → immediate confirm (confidence +10)
  - Soft signal: same status but different response fingerprint from baseline → confidence +3 (catches empty-body responses, stripped data, soft redirects)
- Added `raiseAuthConfidence(delta int)` — accumulates confidence, flips `authBoundaryConfirmed` at threshold of 10

## Fix 3 — Method probing guard

### Problem
`expandMethods` had two guards that were too conservative:
- `ProbeCount > 3` — method probing only fired on the first few probes. Any endpoint discovered later during crawling (JS endpoints, form actions, HTML links) never got method-probed.
- `len(ent.HTTP.Methods) > 1` — bailed out as soon as two methods were known, even if those were just GET and HEAD, leaving POST/PUT/DELETE unprobed.

### Changes

**`knowledge/entity.go`**
- Added `MethodProbed bool` to `EntityState`

**`behavior/expand.go`**
- Replaced both old guards with a single `MethodProbed` flag check
- Every entity now gets exactly one full method sweep (all 7 methods) on its first `Expand` call
- Individual methods are skipped only if already confirmed on that entity
- `MethodProbed` is set to `true` after the sweep so it doesn't repeat

## Fix 4 — Identity probe gating

### Problem
`expandIdentity` fired role confusion and invalid bearer probes on any endpoint
with an IDLike param (`id`, `user_id`, `account` etc.), regardless of whether
there was any evidence of auth on the site. This generated noise requests against
fully public endpoints.

### Changes

**`behavior/expand.go`**
- Removed the IDLike param condition entirely
- Replaced with three evidence-based conditions (any one is sufficient):
  - `hasAuthSignal` — entity has `SigAuthBoundary`, `SigObjectOwnership`, or `SigPossibleIDOR`
  - `hasAuthBehavior` — server issued cookies or auth headers on this endpoint (`ent.HTTP.AuthLikely`)
  - `globalBoundary` — engine confirmed an auth boundary anywhere on the site (`e.authBoundaryConfirmed`)
- Identity probes now only fire when there is real evidence to act on
