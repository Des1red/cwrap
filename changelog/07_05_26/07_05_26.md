# Changelog

```07-05-26``` ```v2.7.5```

## behavior package

### detectPublicAccess — false positive on catchall/home-page responses

**Problem:** Endpoints returning 200 with a fallback or home page were being tagged as `SigPublicAccess` instead of being suppressed.

**Reason:** The only guard was `IsSPAFallback`, which required an exact fingerprint match against the root baseline via `makeFingerprint`. Sites serving slightly different catchall pages (different body, same semantics) slipped through.

**Fix:** Added a secondary fingerprint comparison inside `detectPublicAccess` using `fpString(e.baseStatus, e.baseBody)` against the anonymous response fingerprint. If they match, the tag is suppressed.

---

### Two fingerprinting systems unified

**Problem:** `makeFingerprint` (sha1, struct) and `fpString` (sha256, string) coexisted and could not cross-compare, which caused the type mismatch that surfaced the public access bug.

**Reason:** SPA fallback detection used one system, identity diffs and public access used another. No shared type made comparison impossible without a cast.

**Fix:** Dropped `makeFingerprint` and the `fingerprint` struct entirely. `e.baseFP` is now a `string`. All fingerprint comparisons use `fpString` consistently throughout the package.

---

### expandPathIDs — duplicate neighbor probes on multi-ID paths

**Problem:** A path like `/users/123/posts/456` was generating two batches of neighbor probes — one for each ID segment — when only one batch was intended.

**Reason:** `e.probedPathTemplates[tmpl]` was marked after the segment loop completed, so both ID positions passed the "seen" check before either could claim the template.

**Fix:** Moved the template mark to immediately before the neighbor probe block, inside the loop. The first ID segment claims the template; subsequent segments see it as already marked and skip. Self-probes still run per-segment as intended.

---

### analyzeIDOR — misleading variable name

**Problem:** `noCredDenied` was set to `true` precisely when a credentialed identity was denied — the opposite of what the name implied.

**Reason:** Double-negative naming introduced ambiguity when reading the IDOR signal logic.

**Fix:** Renamed to `credDenied`.

---

### runQueuedProbes — function split

**Problem:** `runQueuedProbes` was a single large function handling request building, identity execution, detection, param classification, reference resolution, identity marking, and analysis.

**Reason:** No separation of concerns made the function hard to read and maintain.

**Fix:** Split into focused helpers with no behavior change:
- `buildProbeRequest` — constructs the probe request from base and probe config
- `executeProbeIdentities` — runs the identity loop, returns responses, statuses, and fingerprints
- `classifyProbeParams` — classifies query and path params after identity execution
- `resolveRef` — picks the reference fingerprint for identity comparison
- `markEffectiveIdentities` — tags identities whose response differed from the reference
- `runProbeAnalyzers` — runs all analyzer passes on the target entity

---

### Run — baseline and session check separated

**Problem:** The baseline request used for `e.baseFP` could include session cookies, meaning the SPA/catchall fingerprint was potentially derived from an authenticated response.

**Reason:** Session detection and baseline fingerprinting were done in a single request, coupling two concerns that need different inputs.

**Fix:** The baseline request is now always sessionless — `e.baseFP` is always a public response fingerprint. Session validity is checked separately via a dedicated `sessionReq` using the LiveSession identity. `extractIdentity` and `captureSession` operate on `sessionResp`, not the baseline.
