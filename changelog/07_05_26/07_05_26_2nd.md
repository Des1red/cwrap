# Changelog

```07-05-26``` ```v2.7.6```

## 1. Refactored queued probe execution

### Summary
Split the large probe loop into smaller behavior-focused functions while preserving the original execution flow.

### Problem
`runQueuedProbes` had too much logic in one function: request construction, identity execution, auth detection, param classification, analyzer execution, learning, and queue propagation were all mixed together.

### Reason
The function was becoming difficult to inspect, modify, and debug safely. Small changes could accidentally affect unrelated behavior.

### Fix
Extracted clear helper functions:

- `buildProbeRequest`
- `executeProbeIdentities`
- `classifyProbeParams`
- `resolveRef`
- `markEffectiveIdentities`
- `runProbeAnalyzers`

Also restored the original behavior guard:

```go
if len(identityStatuses) > 0 {
    e.classifyProbeParams(probe, target)
}
```

This prevents params from being classified when no identity successfully received a response.

## 2. Separated public baseline from live session validation

### Summary
Changed the `Run` flow so the baseline request remains sessionless, while restored cookies are validated using a separate live-session request.

### Problem
The previous stale-session logic checked the public baseline response as if it were a live-session response.

### Reason
`baseReq` is intentionally sessionless and is used for SPA/catchall fingerprinting. A `401` from that request does not prove the stored session is stale.

### Fix
Kept the public baseline request as sessionless:

```go
resp, err := transport.Do(baseReq)
```

Added a separate session validation request:

```go
sessionReq := meta.Apply(cloneRequest(baseReq))
sessionResp, err = transport.Do(sessionReq)
```

If `sessionResp.StatusCode == 401`, stored session cookies are cleared and identities are re-derived.

## 3. Corrected live-session response attribution

### Summary
Ensured that live session identity extraction and session capture use the response from the actual session-applied request.

### Problem
The code was still using the public baseline `resp` when calling:

```go
extractIdentity(ent, meta.Name, resp)
e.captureSession(ent, meta, resp, base.URL)
```

### Reason
`resp` came from a sessionless baseline request, so attributing it to `LiveSession` polluted identity/session state.

### Fix
Changed live-session extraction and capture to use `sessionResp`:

```go
extractIdentity(ent, meta.Name, sessionResp)
e.captureSession(ent, meta, sessionResp, base.URL)
```

## 4. Improved public access detection

### Summary
Added a fingerprint guard to avoid marking SPA/catchall responses as public access.

### Problem
Endpoints returning the same body as the public baseline could be falsely tagged as public endpoints.

### Reason
SPAs often return the same HTML shell for unknown paths, API routes, or frontend routes.

### Fix
Added an anonymous fingerprint comparison:

```go
if fp := probeFP[knowledge.Anonymous]; fp != "" && fp == e.baseFP {
    return
}
```

This suppresses `SigPublicAccess` when the anonymous response is identical to the public baseline.

## 5. Added probe logging to entities

### Summary
Added compact probe logging to `knowledge.Entity` for tagged entities.

### Problem
Reports showed status summaries like `200×12 401×4`, but did not show which identity/probe generated those statuses.

### Reason
Without probe-level evidence, it was harder to understand why a signal or finding was generated.

### Fix
Added:

```go
type ProbeLogEntry struct {
    URL      string
    Method   string
    Reason   string
    Identity string
    Status   int
    FP       string
    Location string
    Count    int
}
```

Added `Entity.ProbeLog` and `Entity.AddProbeLog()` with deduplication by:

- URL
- Method
- Reason
- Identity
- Status
- Location
- Fingerprint

Recorded probe logs inside `executeProbeIdentities()` after reading the response body.

## 6. Added tagged-entity probe log section to full reports

### Summary
Added a new report section for probe logs, limited to entities that have active signals.

### Problem
A full raw probe log would be too noisy, but no probe log made findings harder to audit.

### Reason
The report needs supporting evidence without becoming unreadable.

### Fix
Added:

```go
writeTaggedProbeLog(w, k)
```

The section prints only tagged entities and filters out noisy responses:

```go
func shouldPrintProbeLog(p knowledge.ProbeLogEntry) bool {
    return p.Status != 404 && p.Status < 500

```

Also fixed empty probe-log entity sections by collecting printable rows before printing each entity.

## 7. Fixed probe log URL rendering

### Summary
Updated probe log URL formatting so it matches actual request behavior.

### Problem
The report could show duplicate query params, such as:

```text
/api/orders?user_id=1&user_id=2
```

### Reason
The transport layer used `q.Set()` to replace query values, but `probeLogURL()` manually appended query params.

### Fix
Changed `probeLogURL()` to parse the URL and use `q.Set()` just like transport:

```go
q := u.Query()
for _, p := range req.Flags.Query {
    q.Set(p.Key, p.Value)
}
u.RawQuery = q.Encode()
```

## 8. Fixed analyzer reflection/object-access conflict

### Summary
Changed param behavior analysis so weak stable-structure evidence no longer overwrites stronger object-access/IDOR evidence.

### Problem
A param could end up with conflicting evidence like:

```text
Reflection + ObjectAccess + structure-changes + stable-structure
```

### Reason
`analyzeParamBehavior()` reset current flags on every run, while `ObservedChanges` stayed cumulative.

### Fix
Removed aggressive resets and made strong evidence win:

- Object access beats reflection.
- IDOR/ownership/auth evidence prevents stable-structure from marking a param as reflection.
- Added `isPureReflection()` so analyzers only skip params that are reflection-only and have no stronger evidence.

## 9. Distinguished real input params from JSON response fields

### Summary
Changed analyzer logic to avoid treating JSON response fields as primary exploit parameters when real input params exist.

### Problem
Query-param IDOR findings were sometimes reported against JSON response fields like `id` instead of the real controllable param `user_id`.

### Reason
`ParamJSON` represented response schema fields, but scanner-injected probes could make JSON fields appear controllable.

### Fix
Added helpers:

```go
isRealInputParam()
isResponseDerivedParam()
entityHasRealInputIDParam()
```

Real input params are now:

- query params
- path params
- form params

Scanner-injected params are no longer treated as real discovered input for primary IDOR attribution.

---

## 10. Promoted real input IDs over response-derived IDs

### Summary
Added consolidation logic that moves IDOR/ownership attribution from JSON response IDs to real input IDs when appropriate.

### Problem
A JSON `id` field could still retain `OwnershipBoundary` or `PossibleIDOR` after the real query param was discovered.

### Reason
Analyzer state is sticky. Once JSON `id` was marked, later discovery of `user_id` did not automatically clean up the attribution.

### Fix
Added:

```go
promoteRealInputIDOverResponseID(ent)
demoteResponseDerivedIDORIfRealInputExists(ent)
```

The analyzer pass now runs:

```go
promoteRealInputIDOverResponseID(target)
demoteResponseDerivedIDORIfRealInputExists(target)
```

This keeps response fields as supporting evidence and moves primary findings to the controllable input.

## 11. Added object-access surface analyzer

### Summary
Added a softer analyzer for real input IDs that show access-control behavior without full cross-identity proof.

### Problem
Some endpoints, such as `/api/profile?user_id=1`, showed real object access control through `user_id`, but did not meet the stricter `PossibleIDOR` conditions.

### Reason
A single identity may show both success and denial across different object values. This is useful evidence, but not always enough for a high-confidence horizontal privilege escalation finding.

### Fix
Added:

```go
analyzeObjectAccessSurface(ent, statuses)
```

It marks real input ID params with:

- `LikelyObjectAccess`
- `AuthBoundary`
- `SuspectIDOR`
- `single-identity-object-access-control`

This allows softer reporting such as:

```text
Possible IDOR surface: /api/profile?user_id=1 (param: user_id)
```

without falsely reporting JSON `id` as the exploit parameter.

## 14. Fixed session store directory creation

### Summary
Fixed session persistence so session files can actually be written to disk.

### Problem
Session cookies were not appearing under the config directory.

### Reason
`model.ConfigDir()` created only:

```text
~/.config/cwrap
```

but `session.Save()` wrote to:

```text
~/.config/cwrap/sessions/<host>.json
```

without creating the `sessions` directory.

### Fix
Updated `session.Save()` to create the parent directory:

```go
os.MkdirAll(filepath.Dir(p), 0755)
```

Also changed session file permissions to `0600` because files contain cookies/tokens.

## 15. Improved curl cookie jar to session sync

### Summary
Improved Netscape cookie jar parsing and syncing into the recon session store.

### Problem
Cookies from `cookies.txt` were not reliably synced into `session.Store`, and HttpOnly CSRF cookies could be skipped.

### Reason
The parser did not preserve all useful cookie metadata and skipped `#HttpOnly_` entries in CSRF lookup.

### Fix
Updated `SyncJarToSession()` to:

- load the cookie jar from `~/.config/cwrap/cookies.txt`
- create/update the `session` identity
- preserve domain, path, secure, and HttpOnly metadata
- return `session.Save()` errors

Updated CSRF cookie lookup to handle `#HttpOnly_` lines and match cookies containing `csrf` or `xsrf`.
