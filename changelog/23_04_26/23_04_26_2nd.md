# cwrap — Changelog
`2026-04-23` `v2.4.2`

## Overview
Identity vault implementation, token principal verification, package
restructure, and exploit engine result tracking. All three probes now use
per-identity frozen cookie snapshots with JWT principal verification.

---

## Identity Vault

### Problem solved
`FindIdentityCookies` was returning wrong tokens — all identities were
getting the last-issued session cookies from the shared session block,
not their own frozen snapshots.

### New: `DiscoveredIdentities` on `Knowledge`
```go
type Knowledge struct {
    // ...existing...
    DiscoveredIdentities map[string]map[string]string // name -> cookie name -> value
}
```

### New: Snapshot stored in `addLiveIdentity`
When a new live identity is discovered and its cookie snapshot is frozen,
it is also stored in the knowledge vault:
```go
e.k.DiscoveredIdentities[name] = snapshot
```

### New: `IDENTITY VAULT` section in report
Written at end of full report after entity intelligence:
```
------------------------------------------------
IDENTITY VAULT
------------------------------------------------
  member-uid-1:
    auth_token=eyJ...Alice's actual token...
  member-uid-2:
    auth_token=eyJ...Bob's actual token...
  owner-uid-3:
    auth_token=eyJ...Admin's actual token...
```

Each entry contains the frozen cookie snapshot captured at identity
discovery time — not the last-issued value from any shared session block.

### New: `IdentityVault` on `Report`
```go
type Report struct {
    Target        string
    Entities      []ReportEntity
    IdentityVault map[string]map[string]string
}
```

### New: Vault section parsed in `LoadReport`
Section-aware parser handles `"IDENTITY VAULT"` header, identity name
lines (`"  member-uid-2:"`), and cookie value lines (`"    auth_token=eyJ..."`).

## Token Principal Verification

### New: `tokens` package
Replaces the split `cookies` package. Owns all token-related operations:

- `FindIdentityCookies(r, name)` — reads directly from `IdentityVault`
- `BuildCookieMap(ent, id)` — matches identity cookie names to session values
- `ParseJWTPrincipalUnsafe(token)` — decodes JWT claims without verification
- `FirstCookieValue(jar)` — returns first JWT value from cookie map, fallback to first
- `ResolveTokenPrincipal(id, jar)` — resolves role/uid from identity metadata or JWT

### Fix: Effective identity from token claims
All three probes now verify token principal from vault JWT after metadata
lookup:

```go
role, uid := find.FindIdentityMeta(p.Report, deniedID)
// override with actual token claims — vault token may differ
if tokenRole, tokenUID := tokens.ParseJWTPrincipalUnsafe(tokens.FirstCookieValue(jar));
    tokenRole != "" || tokenUID != "" {
    role, uid = tokenRole, tokenUID
}
```

This surfaces mismatches between the probe identity name and the actual
token principal — e.g. if `member-uid-2` was issued Alice's token during
recon, the output shows `role=member uid=1` rather than silently using
the wrong identity.

## Package Restructure

### Removed: `probe/cookies` sub-package
Merged into `tokens` package alongside JWT parsing utilities.

### New: `tokens` package (`internal/exploit/tokens`)
Single package for all cookie and token operations used by probes.

### New: `replay` package (`internal/exploit/replay`)
Extracted HTTP client and request execution from probe files:
- `NewHTTPClient()` — shared client with 10s timeout, no redirect follow
- `DoGETWithCookies(client, url, jar)` → `*ReplayResult`
- `ReplayResult{StatusCode int, Body []byte}`

### New: `result` package (`internal/exploit/result`)
Tracks probe execution results for reporting:
- `Session{Recon *Report, Steps []Step}`
- `Step` — full probe execution record with phase, kind, identity, status
- Query helpers: `ConfirmedByKind`, `ConfirmedByPhase`, `ByIdentity`, `HasStep`
- `BuildEffectiveIdentity(role, uid)` — formats `member-uid-2` from role/uid
- `BuildPreview(body, max)` — truncates response body for display

### New: `find` package (`internal/exploit/find`)
- `FindCredlessIdentity(ent)` — finds identity with issued-token but no creds
- `FindProtectedEntities(r)` — finds entities with AuthBoundary signal
- `FindIdentityMeta(r, name)` — returns role/uid for named identity

## Probe: Print Helpers

### New: `probe/helpers.go`
Shared print functions used across all three probe files:
- `printIdentityLine(identity, role, uid)` — prints `testing: X (role=Y uid=Z)`
- `printPrimaryIdentity(identity, role, uid)` — prints `identity: X (role=Y uid=Z)`
- `printCookies(prefix, jar)` — prints all cookies with prefix
- `displayIdentity(step)` — formats `identity => effective` when different
- `sortedIdentityKeys(m)` — sorted keys from identity map
- `isSyntheticIdentity(id)` — skips anonymous/fake-admin
- `addStep(session, step)` — nil-safe step recorder

## Result Tracking

### New: All probes record `result.Step`
Each probe execution records a step with full context:
```go
result.Step{
    Phase:             result.Stage1,
    Kind:              result.KindCredentialless,  // or KindIDOR, KindOwnership
    EntityURL:         p.Entity.URL,
    TargetURL:         ent.URL,
    Identity:          credlessID.Name,
    EffectiveIdentity: effective,
    Role:              role,
    UserID:            uid,
    StatusCode:        res.StatusCode,
    Confirmed:         res.StatusCode == 200,
    Summary:           "...",
    Response:          preview,
}
```

Under `--debug`, exploit prints recorded and confirmed step counts:
```
Recorded steps: 4
Confirmed steps: 2
```
