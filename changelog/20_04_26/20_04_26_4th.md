# cwrap recon — recon engine accuracy fixes
`20/04/26` `v2.3.1`

## Overview
Four bugs in the recon engine's accuracy and probe efficiency were identified
and fixed. The fixes address an infinite enumeration loop, corrupted baseline
data poisoning analyzers, false auth boundary signals on every entity, and
wasteful method sweeps against probe-generated URL variants.

---

## Bug #1 — Path ID enumeration infinite loop

### Problem
`expandPathIDs` generates path variant probes (e.g. `/api/users/0`,
`/api/users/1`) when it detects a numeric or UUID segment in a URL. Each
variant became a new entity with fresh state (`PathIDProbed = false`),
causing `expandPathIDs` to fire again on every variant and chain
indefinitely. The engine would spiral into thousands of probes against
incrementing IDs until killed manually.

### Fix
Added engine-level `probedPathTemplates map[string]struct{}` that normalizes
any path-ID URL to a template before probing (e.g. `/api/users/1312` →
`/api/users/{id}`). `expandPathIDs` checks the template on entry and bails
if already seen. First entity through wins; all subsequent variants are
silently skipped.

### Changes

**`behavior/engine.go`**
- Added `probedPathTemplates map[string]struct{}` field
- Initialized in `New()`

**`behavior/expand.go`**
- Added `pathTemplate(u *url.URL) string` helper
- Added template check at top of `expandPathIDs` before any probe generation

## Fix — Injected param noise in reports

### Problem
`expandDiscovery` injects params like `api`, `id`, `user`, `page`, `q` as
candidate query params. These accumulated `IdentityAccess`/`IdentityDenied`
maps that mirrored entity-level auth behavior rather than anything the param
itself caused. They were passing the `paramHasEvidence` check and cluttering
the report with meaningless entries.

### Fix
In `paramHasEvidence()`, added an early-exit guard before the access/denied
map checks: if the param is injected-only, return false immediately. Injected
params now only appear in the report if they carry real structural evidence
(ObjectAccess, AuthBoundary, ObservedChanges, etc.).

### Changes

**`report/helpers.go`**
- `paramHasEvidence`: added `if p.InjectedOnly() { return false }` before
  access/denied map checks

## Bug #2 — Auth boundary logic gaps

### Problem
Four separate gaps caused incorrect or missing auth boundary signals:

1. **Corrupted path param baselines.** `storeResponse` was using the global
   `baseStatus`/`baseBody` (the root URL's response — often a 303 redirect)
   as the baseline entry for path param probes. Analyzers were comparing
   probe responses against redirect bodies.

2. **Broad-brush `AuthBoundary` tag.** `analyzeMethods` was tagging every
   entity with `AuthBoundary` whenever it had multiple identities and
   state-changing methods — regardless of whether access was actually gated.
   This produced `AuthBoundary: 18` on all 18 entities.

3. **No entity-level auth boundary detection without param evidence.**
   After removing the broad-brush tag, entities with no probed params
   (e.g. `/home`, `/admin`) had no path to `AuthBoundary`.

4. **`ObjectAccess` lost on path params.** Per-probe `responses` maps only
   ever held 1 value per param, so `analyzeParamBehavior`'s `len(byVal) < 2`
   check always bailed. The corrupt baseline injection was accidentally
   providing a second value to diff against — removing it broke structural
   analysis.

### Fix

1. Skip baseline injection in `storeResponse` for path params. Path ID
   probes generate enough variant values to compare without a baseline.

2. Removed the broad-brush `AuthBoundary` tag from `analyzeMethods`.

3. Added `detectAuthBoundary(ent, identityStatuses)` — tags `AuthBoundary`
   when a credentialed identity got 200 and an uncredentialed identity was
   denied. Called in `runQueuedProbes` directly from `identityStatuses`,
   same pattern as `detectRoleBoundary`.

4. Added `AccumResponses`/`AccumStatuses` to `knowledge.Entity`. `storeResponse`
   mirrors all writes into these maps. For path params, accumulation targets
   the **source entity** (the one that ran `expandPathIDs`) via
   `probe.SourceURL`, so all variant responses accumulate together.
   `analyzeParamBehavior` now runs on `target.AccumResponses`, and also runs
   on the source entity after each path ID probe.

### Changes

**`internal/recon/knowledge/entity.go`**
- Added `AccumResponses map[string]map[string]map[string][]byte`
- Added `AccumStatuses  map[string]map[string]map[string]int`
- Both initialized in `NewEntity()`

**`internal/recon/knowledge/probe.go`**
- Added `SourceURL string` to `Probe` struct

**`behavior/expand.go`**
- `expandPathIDs`: sets `SourceURL: ent.URL` on all generated probes

**`behavior/store_response.go`**
- Path param baseline injection skipped (`isPathParam` guard)
- Accumulated map writes added after per-probe writes
- Path params accumulate on source entity (`kg.Entity(probe.SourceURL)`)
- Parameter renamed from `k` to `kg` to avoid shadowing loop variable
- Call site updated to pass `e.k`

**`behavior/analyze.go`**
- Removed broad-brush `AuthBoundary` tag from `analyzeMethods`

**`behavior/detect.go`**
- Added `detectAuthBoundary(ent *knowledge.Entity, identityStatuses map[string]int)`

**`behavior/runner.go`**
- `analyzeParamBehavior` now called with `target.AccumResponses`
- `detectAuthBoundary` called after `detectRoleBoundary`
- After path ID probes, `analyzeParamBehavior` also called on source entity
- `storeResponse` call updated with `e.k` argument

## Bug #3 — Method probe guards

### Problem
`expandMethods` fired on every entity including path-variant entities like
`/api/users/0` and `/api/posts/-1`. These are probe-generated URLs that
likely 404 or are invalid ID space — running a full 7-method sweep against
them was wasteful. `/api/posts/0` was hitting 18 probes when only GET was
warranted.

Additionally, organically discovered entities (e.g. `/api/users/1` linked
from `/home`) could be incorrectly marked as path variants if a path ID probe
hit them before the HTML discovery edge was registered.

### Fix
Added `IsPathVariant` and `OrganicallyDiscovered` flags to `EntityState`.
Path ID probes set `IsPathVariant = true` on their target if the target has
not yet been organically discovered. `expandMethods` skips the method sweep
on path variant entities. `Knowledge.AddEdge` marks destination entities as
`OrganicallyDiscovered = true` and clears `IsPathVariant` whenever an HTML
or form edge is recorded — ensuring correct behavior regardless of probe
ordering.

### Changes

**`internal/recon/knowledge/types.go`**
- Added `IsPathVariant bool` to `EntityState`
- Added `OrganicallyDiscovered bool` to `EntityState`

**`internal/recon/knowledge/knowledge.go`**
- `AddEdge`: marks destination entity `OrganicallyDiscovered = true` and
  `IsPathVariant = false` for `EdgeDiscoveredFromHTML` and `EdgeFormAction`

**`behavior/runner.go`**
- Sets `target.State.IsPathVariant = true` when `probe.Reason == ReasonPathIDProbe`
  and `!target.State.OrganicallyDiscovered`

**`behavior/expand.go`**
- `expandMethods`: returns early (after setting `MethodProbed = true`) if
  `ent.State.IsPathVariant`

## Bug #4 — Premature identity expansion

### Problem
`expandIdentity` was firing on every entity in the enumeration chain,
producing identity probe floods against hundreds of `/api/users/N` variants.

### Fix
Resolved as a direct consequence of Bug #1. Once the infinite enumeration
loop was killed, identity probes correctly fire only on the bounded set of
entities that warrant them. No independent code change required.
